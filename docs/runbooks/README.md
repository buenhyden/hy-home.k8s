# Runbooks

- **Status**: Approved
- **layer:** meta

**Overview (KR):** 인프라 작업, 장애 대응 및 배포 프로세스를 실행하기 위한 구체적인 운영 절차서 모음입니다.

This directory is the **exclusive home** for all executable operational procedures, incident response playbooks, and deployment guides.

## 1. Necessity and Purpose

This directory is necessary to separate active infrastructure commands from static project knowledge. If it tells a human or an automated system "how to do something to the infrastructure", it belongs here securely.

- **Operations Anchor**: AI DevOps Agents treat this folder as the absolute truth for deployment, rollbacks, and incident mitigation steps.

## 2. Strict Boundary Rule

**Never place these files in `docs/runbook`.**
Placing operational logs inside the long-term documentation hub leads to clutter and confusion. The DevOps Agent expects to find all operational scripts and procedures directly in this root-level `/runbooks/` folder.

## 3. Required Content & Templates

Every runbook generated by humans or the DevOps Agent MUST be instantiated using the `templates/runbook-template.md` base.

- **Required Content**: Bash commands, failover steps, required ENV vars, and rollback procedures.
- **Sub-folders (Optional)**: `incidents/`, `postmortems/`, `services/`.

## 4. Agent Guidelines & Standards

AI Agents generating or modifying service runbooks inside this directory MUST adhere to `.agent/rules/0381-runbooks-oncall.md`.
Specifically, all service runbooks must conform to the structure defined in `REQ-RBK-STR-01` (including Dashboards, SLOs, Alerts, and Rollbacks). Any automated deployment rules will block production releases if a compliant runbook does not exist for a given service.
